# Noctilink MVP 范围与章节结局规则 v0.1

> 项目中文名：夜链  
> 项目英文名：Noctilink  
> 文档用途：固定 v0.1 阶段的最小可行范围、章节结局规则和后续设计边界。  
> 建议路径：`docs/11_mvp_scope_and_ending_rules.md`

---

## 1. 文档目的

本文档用于固定 Noctilink 当前阶段的 MVP 范围与章节结局规则，避免后续剧情设计、系统设计或 Codex 生成内容时过早引入过多视角、商业化、后端、复杂因果图和多章节分支继承。

当前阶段的目标不是做完整商业化游戏，而是验证核心玩法：

```text
单视角文字冒险
属性判定
章节内多结局
主线推进结局
结局图鉴
本地存档
后续多视角因果链预留
```

---

## 2. 当前阶段总原则

Noctilink v0.1 MVP 采用“先验证核心体验，再扩展多视角”的策略。

当前阶段只做：

```text
序章 + 第一章
A 视角单线可玩
属性判定
多个章节闭合结局
一个主线推进结局
只有主线推进结局解锁第二章
```

当前阶段不做：

```text
B/C 可玩视角
商业化
付费解锁
广告
云存档
完整后端
复杂因果图
多章节多结局继承
```

---

## 3. MVP 定义

MVP 是 Minimum Viable Product，即“最小可行产品”。

在 Noctilink 中，MVP 指的是：

```text
用最小内容量和最小系统复杂度，
验证“都市异常文字冒险 + 属性判定 + 章节多结局 + 主线推进”的体验是否成立。
```

MVP 不是最终游戏，也不是完整试玩版。它是一个小而完整的验证版本。

---

## 4. MVP 内容范围

### 4.1 章节范围

```text
序章
第一章
```

序章负责建立：

```text
都市异常氛围
A 视角代入
基础操作
核心属性概念
第一章事件动机
```

第一章负责验证：

```text
剧情选择
属性判定
结局分歧
闭合结局收集
主线推进结局
本地存档
```

### 4.2 视角范围

```text
仅 A 视角可玩
```

B/C 视角在 MVP 中暂不开放为可玩内容。

可以保留：

```text
B/C 相关伏笔
未知人物痕迹
未解释事件
后续多视角扩展接口
```

但不能要求 MVP 玩家必须操作 B/C 才能理解第一章的基本故事。

### 4.3 玩法范围

MVP 必须包含：

```text
剧情文本展示
选项选择
属性判定
属性变化
世界状态变化
结局触发
结局图鉴
本地存档
章节解锁判断
```

MVP 可选但不强制包含：

```text
简单线索系统
简单结局提示
简单回溯功能
简单异常档案
```

当前阶段明确不做以下功能：

```text
B 视角可玩
C 视角可玩
付费视角
章节买断
广告提示
商城

复杂资源管理
```

这些功能可以在后续版本中扩展，但不能影响 MVP 的实现节奏。

## 6. 章节结局规则

### 6.1 核心规则

Noctilink 不采用“所有结局都进入下一章”的结构。

每章允许多个结局，但结局分为两类：

```text
章节闭合结局
主线推进结局
```

大部分结局是章节闭合结局。只有主线推进结局可以解锁下一章。

### 6.2 为什么不让所有结局进入下一章

如果第一章的每个结局都要进入第二章，后续章节会迅速分支爆炸。

错误结构：

```text
第一章结局 A -> 第二章状态 A
第一章结局 B -> 第二章状态 B
第一章结局 C -> 第二章状态 C
第一章结局 D -> 第二章状态 D
```

这种结构到第三章会变得难以维护。

正确结构：

```text
第一章多个结局
├── 闭合结局 1：收录，不解锁第二章
├── 闭合结局 2：收录，不解锁第二章
├── 闭合结局 3：收录，不解锁第二章
└── 主线推进结局：收录，并解锁第二章
```

---

## 7. 结局类型定义

### 7.1 章节闭合结局

章节闭合结局指当前路线在本章内结束。

它可以是坏结局，也可以不是坏结局。

常见类型包括：

| 类型 | 说明 | 是否解锁下一章 |
|---|---|---|
| 坏结局 | A 死亡、崩溃、失控、被捕等 | 否 |
| 退场结局 | A 活下来，但离开事件核心 | 否 |
| 误解结局 | A 相信了错误解释，以为事件结束 | 否 |
| 污染结局 | A 接触到异常，但方向错误或被污染影响 | 否 |
| 支线闭合结局 | A 达成局部目标，但没有进入主线状态 | 否 |

章节闭合结局的作用：

```text
进入结局图鉴
补充世界观
提示错误方向
展示异常后果
提供重玩动机
为后续多视角埋伏笔
```

章节闭合结局不应让玩家感觉“白玩了”。每个闭合结局都应至少提供一种信息价值。

### 7.2 主线推进结局

主线推进结局是本章唯一可以解锁下一章的结局。

它代表玩家抵达了后续章节所需的关键状态。

主线推进结局必须：

```text
进入结局图鉴
解锁下一章
写入主线锚点状态
成为后续章节的唯一继承基础
```

## 8. “假结局”命名规则

内部设计中不建议使用“假结局”作为正式名称。

推荐使用：

```text
章节闭合结局
偏离结局
退场结局
误解结局
污染结局
支线结局
主线推进结局
```

玩家端统一显示为“结局”。

不要在玩家界面直接提示：

```text
这是假的结局
```

更推荐提示：

```text
你抵达了一个结局。
这条路线已经闭合，但仍有未解开的因果。
```

---

## 9. 第一章结局数量建议

第一章建议控制在：

```text
3-5 个章节闭合结局
1 个主线推进结局
```

不要一开始设计过多结局。

推荐第一章结局分布：

| 结局类型 | 数量建议 | 作用 |
|---|---:|---|
| 坏结局 | 1-2 | 展示危险、异常、失败代价 |
| 退场结局 | 1 | 让角色活下来但离开主线 |
| 误解结局 | 1 | 让玩家意识到信息不足 |
| 污染结局 | 0-1 | 展示污染机制 |
| 主线推进结局 | 1 | 解锁第二章 |

---

## 10. 主线推进结局设计原则

主线推进结局只有一个，但不应只有一条死板路线。

推荐规则：

```text
主线推进结局唯一。
抵达主线推进结局的路径可以不唯一。
```

也就是说，主线推进结局不应该要求玩家必须按固定选项序列游玩，而应该要求玩家满足关键条件。

示例条件：

```text
获得至少 2 条关键线索
理智高于最低阈值
污染值处于可控区间
最终节点选择继续接近真相
```

玩家可以通过不同方式满足条件：

```text
靠洞察发现线索
靠污染值看到异常线索
靠消耗体力抵达关键地点
靠保留理智避免误判
```

## 11. 属性判定与结局关系

属性判定要服务于结局结构，而不只是简单成功或失败。

当前 MVP 可使用的核心属性：

```text
理智
体力
洞察
污染值
```

暂不强制引入金钱、好感、复杂资源管理。

属性与结局关系示例：

| 属性状态 | 可能导致 |
|---|---|
| 理智过低 | 误判、崩溃、污染类闭合结局 |
| 体力不足 | 无法抵达关键地点，进入退场结局 |
| 洞察不足 | 错过关键线索，进入误解结局 |
| 污染值过低 | 看不到异常层信息，无法进入主线推进路径 |
| 污染值过高 | 看到真相但无法保持自我，进入污染结局 |

注意：污染值不应单纯等于坏东西。它可以带来信息优势，但也会增加失控风险。

## 12. 主线锚点状态

### 12.1 定义

主线锚点状态是主线推进结局写入的标准状态，用于后续章节继承。

后续章节不继承所有闭合结局的状态，只继承主线推进结局对应的主线锚点。

### 12.2 示例

```json
{
  "chapter": "ch01",
  "canonEnding": "ch01_a_canon_ending",
  "unlocks": ["ch02"],
  "canonState": {
    "A.status": "alive",
    "A.knows_anomaly_exists": true,
    "world.ch01_canon_reached": true,
    "clues.core_signal": true
  }
}
```

### 12.3 设计原则

主线锚点应该只保存后续章节真正需要的状态。

不要把第一章所有细节都继承到第二章。

推荐只继承：

```text
主角是否存活
主角是否确认异常存在
关键线索是否获得
核心世界状态是否改变
章节是否已被主线推进
```

## 13. 结局图鉴规则

MVP 应包含最小结局图鉴。

结局图鉴记录：

```text
结局 ID
结局名称
结局类型
所属章节
所属视角
是否主线推进
是否解锁下一章
触发时间
简短描述
已获得线索
偏离原因提示
```

结局图鉴的作用：

```text
给非主线结局价值
引导玩家重玩
提示玩家还有未解开的因果
为未来多周目和多视角系统预留空间
```

## 14. MVP 最小数据需求

### 14.1 结局字段

剧情节点或结局配置中应支持：

```yaml
ending:
  id: ch01_a_ending_03
  name: "退场结局：雨夜离城"
  type: closed
  category: retreat
  unlock_next_chapter: false
  archive: true
  hint: "你活了下来，但错过了继续追查的机会。"
```

主线推进结局示例：

```yaml
ending:
  id: ch01_a_canon_ending
  name: "主线推进：确认异常"
  type: canon
  category: mainline
  unlock_next_chapter: true
  archive: true
  unlocks:
    - ch02
  canon_state:
    A.status: alive
    A.knows_anomaly_exists: true
    world.ch01_canon_reached: true
    clues.core_signal: true
```

### 14.2 存档字段

MVP 存档至少包含：

```text
当前章节
当前节点
当前视角
角色属性
世界状态
选择历史
已解锁结局
已解锁章节
主线锚点状态
内容版本号
```

暂不需要保存：

```text
付费权益
广告奖励
云存档 ID
订单信息
复杂多周目数据
```

## 15. MVP 验收标准

MVP 完成时，应满足以下条件：

```text
可以从序章进入第一章
可以完成 A 视角第一章
玩家选择会影响属性
属性会影响可选选项或结局
第一章至少有 3 个闭合结局
第一章有且只有 1 个主线推进结局
闭合结局进入结局图鉴但不解锁第二章
主线推进结局进入图鉴并解锁第二章
本地存档可以保存当前进度
重新进入游戏可以恢复进度
```

可选验收：

```text
结局图鉴中显示偏离原因提示
达成闭合结局后可以快速回到关键节点
StoryTool 可以检查结局配置是否合法
```

## 16. StoryTool 对本规则的校验要求

后续 StoryTool 应至少校验：

```text
每章必须至少有 1 个主线推进结局
每章主线推进结局最多 1 个
闭合结局不能解锁下一章
主线推进结局必须配置 canon_state
主线推进结局必须配置 unlocks
结局 ID 不能重复
所有结局必须进入结局图鉴
unlock_next_chapter 与 type 必须一致
```

示例规则：

```text
type = closed  -> unlock_next_chapter 必须为 false
type = canon   -> unlock_next_chapter 必须为 true
```

## 17. 后续扩展预留

当前 MVP 只做 A 视角，但规则要允许后续扩展到多视角。

未来 B/C 视角可沿用同样结构：

```text
每个视角都可以有多个闭合结局
每个视角都可以有一个视角推进结局
章节整体最终只生成一个主线锚点状态
```

未来多视角版本应避免：

```text
A 结局 x B 结局 x C 结局无限组合继承
```

推荐未来结构：

```text
A 视角推进状态
B 视角推进状态
C 视角推进状态
↓
章节总主线锚点
↓
下一章
```

## 18. 当前拍板方案

Noctilink v0.1 MVP 正式采用以下规则：

```text
内容：序章 + 第一章
视角：仅 A 视角可玩
玩法：剧情选择 + 属性判定 + 多结局
结局：多个章节闭合结局 + 一个主线推进结局
解锁：只有主线推进结局解锁第二章
继承：后续章节只继承主线锚点状态
商业化：暂不考虑
后端：暂不考虑
重点：验证单视角章节多结局与主线推进体验
```
