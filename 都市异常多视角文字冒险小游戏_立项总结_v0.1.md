# 都市异常多视角文字冒险小游戏：立项总结文档

- 版本：v0.1
- 记录日期：2026-05-12
- 用途：项目立项留档、后续剧情与系统设计、Codex 生成内容的基础上下文

> 本文档为当前已确认的剧情逻辑与基本玩法总结，不包含具体剧情样例。

## 1. 项目定位

本项目定位为“都市异常题材 · 多主角切换 · 像素风文字冒险 RPG 小游戏”。它不是传统地图探索 RPG，而是以剧情选择、属性判定、角色关系、生存管理、多视角因果链和多结局为核心体验的文字冒险游戏。

项目适合以微信小游戏或类似小程序游戏平台为主要发布目标，采用竖屏体验、轻量美术资源和连载式章节更新方式。

> 核心定义：玩家通过不同主角视角，在同一条时间线中理解并改写事件因果。

| 项目项 | 当前结论 |

## 3. 核心卖点

本作的核心卖点是“多主角因果链”。多个主角位于同一条时间线中行动，不同角色的选择会改变世界状态，并影响其他角色后续的剧情路线、可选行动、生死状态和结局。

付费视角的价值不应只是补充剧情，而是让玩家接管另一个角色，理解免费线中无法解释的事件，并有机会改写关键因果。

- 同一事件可以被不同主角以不同信息量、不同立场、不同风险重新理解。
- 玩家在免费视角中看到的未知结果，可在付费视角或多周目中被解释和改写。
- 坏结局不是单纯失败，而是收集线索、理解因果、推动多周目探索的一部分。

## 4. 世界观方向

世界观建议采用“都市异常污染”方向：城市表面维持日常秩序，但内部正在出现无法解释的异常事件。异常可以表现为记忆篡改、空间错位、身份替换、组织封锁、异常病例、失踪事件、现实规则改变等。

污染值不只是负面状态，也可以是探索隐藏真相的钥匙。污染较低时玩家保持安全与清醒，但可能错过异常层信息；污染较高时可以看到隐藏线索和特殊选项，但会带来失控、幻觉、关系恶化和坏结局风险。

- 整体氛围：现实感、压迫感、悬疑感、异常感、生存危机、信息不完整。
- 叙事重点：玩家逐步意识到城市日常背后存在另一层规则。
- 多周目重点：第一周目看到事件表象，后续周目逐步获得解释与改写能力。

## 5. 多主角叙事结构

建议第一季控制在 3 个主角以内。多主角不应只是换人称重复同一剧情，而应分别承担不同叙事职责。

> 免费主角线必须完整。付费视角可以扩展真相、改变结局、解锁真结局，但不能让免费玩家觉得只玩到了残缺版本。

| 视角 | 商业定位 | 叙事功能 | 玩法价值 |
| --- | --- | --- | --- |
| 主角 A | 免费主线视角 | 提供基础剧情、代入感、普通结局。 | 让玩家完整体验核心玩法与主要悬念。 |
| 主角 B | 付费扩展视角 | 解释 A 线中不明原因的事件。 | 提供关键行动改写能力，改变部分世界状态。 |
| 主角 C | 后期付费 / 隐藏真相视角 | 揭示污染、组织、实验、城市真相等深层信息。 | 解锁多周目隐藏节点和真结局条件。 |

## 6. 商业化结构

当前商业化方向为：一个主角剧情免费，其他角色视角收费，提示广告辅助。此结构与多视角因果链高度匹配。

付费内容应提供真实的新信息、新选择、新因果改写点和新结局，而不是只把同一剧情换个角度复述。

- 不建议强制插屏广告。
- 不建议用广告直接改写结局。
- 提示广告应提供“模糊线索”或“风险提示”，不要直接给正确答案。

| 内容类型 | 建议定位 |
| --- | --- |
| 免费主角线 | 完整序章与前期章节，提供一个可独立成立的普通体验。 |
| 付费角色视角包 | 解释未知因果、提供改写关键事件的能力、增加结局路线。 |
| 整季通行证 | 适合包含多个角色视角、后续章节、隐藏档案或番外。 |
| 提示广告 | 用于关键选择提示、坏结局原因提示、模糊线索提示。 |

## 7. 基本玩法循环

基础玩法循环应保持短、清晰、可重复。玩家进入章节后阅读剧情，做出选择，选择带来属性变化、关系变化、线索变化和世界状态变化，最终进入后续分支、坏结局、普通结局或隐藏结局。

1. 进入章节或时间线节点。
2. 阅读当前主角视角下的剧情。
3. 根据当前属性、线索、资源和关系做选择。
4. 选择写入属性变化、角色关系变化和世界状态变化。
5. 后续节点读取这些状态，生成不同剧情结果。
6. 章节结束后展示结局、因果图、档案解锁或多周目线索。

## 8. 数值与资源系统

已确定存在理智、体力、金钱、洞察、好感、污染值等数值。第一版不宜过度复杂，建议玩家界面只展示核心属性，隐藏变量用于剧情判定。

> 好感不等于信任。某角色可以喜欢主角，但不相信主角的判断；也可以信任主角能力，但情感上保持距离。

| 属性 | 主要用途 |
| --- | --- |
| 理智 | 影响异常承受能力。过低可能触发崩溃、误判、幻觉或坏结局。 |
| 体力 | 影响逃跑、调查、对抗、受伤后的行动能力。 |
| 金钱 | 影响购买道具、交通、情报、治疗、临时避难等行为。 |
| 洞察 | 影响识破谎言、发现隐藏线索、解锁特殊选项。 |
| 污染值 | 影响能否看到异常层信息，同时带来失控和坏结局风险。 |
| 好感 / 信任 | 影响角色是否帮助玩家、透露信息、配合行动、牺牲或背叛。 |

## 9. 失败与坏结局设计

坏结局是本作的重要内容，不应只作为失败惩罚。坏结局应帮助玩家理解因果链、补全世界观、解锁多周目线索，并刺激玩家重新选择。

- 坏结局来源：理智过低、体力耗尽、污染值过高、金钱不足、好感不足、信任错误对象、错过关键线索、时间管理失败、其他主角行动造成连锁后果。
- 坏结局反馈：应让玩家能大致理解失败原因，但不要完全暴露正确路线。
- 坏结局价值：可解锁档案、记忆碎片、因果图节点、隐藏选项提示或多周目信息。

## 10. 跨主角因果链系统

本作的核心机制是“世界状态”。每个主角的关键行动会写入世界状态，其他主角的剧情节点会读取这些状态，从而产生不同结果。

系统设计上不应把所有剧情写死成海量分支，而应把关键影响抽象成可读写的状态变量。

> 设计目标：让玩家感觉“我在一个角色视角中的选择，确实改变了另一个角色的命运”。

| 状态类型 | 示例方向 |
| --- | --- |
| 场景状态 | 门是否打开、电力是否中断、道路是否堵塞、监控是否关闭、火灾是否发生。 |
| 角色状态 | 角色是否存活、是否受伤、是否被捕、是否信任玩家。 |
| 时间状态 | 是否提前抵达、是否延迟通知、某区域是否已经封锁。 |
| 情报状态 | 是否知道地点、是否掌握身份、是否发现秘密。 |
| 污染状态 | 区域污染等级、角色污染程度、是否看见异常层世界。 |

## 11. 默认行动脚本

由于存在免费视角和付费视角，非当前主角不能静止不动。未解锁或未操作的主角必须按照默认行动脚本推进剧情。

当玩家只玩免费视角时，其他角色按默认路线行动；当玩家解锁其他视角后，可以在关键节点接管角色，改写默认行动。

- 每个主角都需要默认行动路线。
- 每个主角都需要默认结局倾向。
- 每个主角都需要可被玩家改写的关键节点。
- 默认行动造成的结果应能在因果图中被逐步揭示。

## 12. 章节与多周目结构

章节结构确定为分支章节和多周目解锁。为控制复杂度，建议采用“主线章节制 + 局部分支 + 关键变量继承 + 多周目隐藏节点”的结构。

不建议做无限树状分支，应采用可控分支并在章节末尾收束到少量大状态。

- 可继承内容：结局图鉴、人物档案、异常档案、关键记忆碎片、已发现线索、已解锁因果节点。
- 不建议大量继承属性数值，否则会显著增加平衡难度。

| 阶段 | 玩家体验 |
| --- | --- |
| 第一周目 | 看到表层事件和有限信息，体验免费主角完整故事。 |
| 第二周目 | 解锁新视角、记忆碎片、隐藏选项，解释部分未知因果。 |
| 第三周目及以后 | 理解更深层真相，主动改写关键因果，尝试达成隐藏结局或真结局。 |

## 13. 美术与 UI 方向

美术风格确定为纯像素 UI、像素头像、像素背景。由于本作不是大地图探索游戏，美术重点应放在阅读体验、界面质感、角色辨识度、场景氛围和异常演出上。

- 核心 UI：剧情阅读界面、选择界面、属性状态栏、角色关系界面、线索/道具界面、结局图鉴、人物档案、异常档案、章节选择界面。
- 特色 UI：因果图界面、多周目时间线界面、未知节点展示界面。
- 资源重点：像素 UI 框、头像表情、场景背景、道具图标、线索图标、结局图标、因果节点图标。

## 14. MVP 范围建议

第一版目标不是完成整季，而是验证“文字冒险 + 多视角因果链 + 角色视角付费”是否成立。推荐 MVP 范围如下。

| 模块 | 建议规模 |
| --- | --- |
| 章节 | 序章 + 2 个正式章节。 |
| 主角 | 3 个主角，其中 1 个免费视角，1 个付费视角试做，1 个隐藏视角预埋。 |
| 主要 NPC | 5-6 个。 |
| 剧情节点 | 35-50 个。 |
| 重要选择 | 20-30 个。 |
| 属性判定 | 10-15 次。 |
| 线索 / 道具 | 15-25 个。 |
| 坏结局 | 5-8 个。 |
| 阶段结局 | 1-2 个。 |
| 像素背景 | 10-15 张。 |
| 像素头像 | 8-12 个。 |
| 首轮体验时长 | 40-90 分钟。 |

## 15. 开发优先级

第一阶段开发重点不是大量写剧情，而是先实现可持续扩写的系统闭环。

1. 剧情节点系统。
2. 选择分支系统。
3. 属性变化系统。
4. 世界状态变量系统。
5. 多主角默认行动脚本。
6. 因果节点记录与展示。
7. 结局判定。
8. 存档 / 读档。
9. 章节解锁。
10. 付费视角解锁。
11. 提示广告入口。

## 16. 剧情数据结构需求

为了便于后续连载、维护和用 Codex 辅助生成内容，剧情必须结构化。每个剧情节点至少应包含以下字段。

- 节点 ID。
- 章节 ID。
- 时间点。
- 当前主角。
- 正文。
- 可选选项。
- 选项条件。
- 属性影响。
- 世界状态读取。
- 世界状态写入。
- 跳转节点。
- 结局触发条件。
- 是否属于付费视角。
- 是否属于隐藏节点。

## 17. 设计边界与风险控制

该项目最容易失控的地方是多主角、分支、多周目和连载内容叠加后导致剧情量爆炸。因此必须提前设定边界。

- 第一版暂不建议加入大地图自由探索、复杂战斗系统、抽卡系统、多人联网、大量角色养成、开放世界结构。

| 规则 | 建议 |
| --- | --- |
| 主角数量 | 第一季最多 3 个主角。 |
| 强因果节点 | 每章最多 3 个。 |
| 生死级变量 | 每章最多 1 个角色生死级变量。 |
| 章节收束 | 每章结尾必须收束到 2-3 个大状态。 |
| 付费视角价值 | 每个付费视角至少提供 1 个关键因果改写点。 |
| 坏结局反馈 | 每个坏结局必须能反推一个失败原因。 |
| 免费内容完整性 | 免费主角线必须具备完整体验。 |

## 18. Codex / Markdown 留档建议

建议必须保留一份 Markdown 版本。Word 适合归档、展示和给非技术人员阅读；Markdown 更适合作为 Codex、脚本、版本管理和后续内容生成的源文档。

后续如果使用 Codex 生成剧情、节点数据、角色设定、配置文件或开发任务，Markdown 文档可以直接放入项目仓库，例如 docs/game_design.md，并在 AGENTS.md 中引用它作为长期上下文。

- Markdown 方便 diff，能清楚看到每次设定变更。
- Markdown 容易被 Codex、编辑器、脚本和 CI 工具读取。
- Markdown 可以拆成多个设计文档：world、characters、story_nodes、systems、monetization 等。
- Word 用作对外版本或阶段性归档；Markdown 用作持续迭代的“源文件”。

## 20. 最终定义

本项目是一款都市异常题材的多视角因果链文字冒险小游戏。游戏采用像素 UI、像素头像和像素背景作为美术包装，以剧情选择、属性判定、角色好感、生存管理和多结局为核心玩法。

玩家将先通过一个免费主角视角体验完整基础剧情，再通过付费解锁其他主角视角，理解并改写同一时间线中的关键因果。不同主角的选择会改变世界状态，并影响其他主角的路线、生死、线索、信任关系和结局。

坏结局不是单纯失败，而是补全真相、解锁多周目和推动玩家理解因果链的重要内容。项目采用连载式章节更新，商业化以免费主角线、付费角色视角、章节买断和提示广告为主。
